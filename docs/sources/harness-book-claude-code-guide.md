## 先给一个判断

这本书真正想讲的，不是 `Claude Code` 有多少功能，而是一个会写代码的模型进入终端、仓库、权限系统和团队流程之后，系统凭什么还能维持边界、连续性和后果控制。

所以它讨论的核心不是“提示词怎么写更好”，而是 `Harness Engineering`:

- 控制面怎样组织
- 主循环怎样持续运行
- 工具和权限怎样被约束
- 上下文怎样治理
- 错误怎样恢复
- 多 agent 怎样验证彼此
- 团队经验怎样沉淀成制度

如果你已经对 Claude Code 的表层能力不陌生，这本书会把你往上带一层，从“会用”进入“看懂这类系统为什么能工作”。

## 适合谁读

1. 在用 `Claude Code`，但还没形成系统心智模型的人
2. 想做 `AI coding agent`、`agent runtime` 或内部开发助手的人
3. 需要思考权限、恢复、审计、治理和团队落地的人
4. 想从“Prompt Engineering”走向“运行时工程”的人

## 这本书最值得抓住的 6 个观点

1. `Harness Engineering` 不是 `Prompt Engineering` 的加大版，而是运行时控制结构。
2. Prompt 在这类系统里更像控制面，而不是聊天输入框。
3. 模型犯错不是异常，而是运行时常态，所以恢复机制必须内建。
4. 权限系统不是为了防模型笨，而是为了防行为后果失控。
5. 多 agent 不是“越多越强”，而是分工、验证和责任边界的重新设计。
6. 真正能落地的不是个人技巧，而是可复用、可传承、可审计的团队制度。

## 章节地图

### 序言：Harness、终端与工程约束

开篇先把问题摆正。模型一旦进入真实工程环境，面对的就不是问答任务，而是带后果的执行任务。终端、仓库、文件系统、权限和团队流程，会把“回答质量”问题迅速升级成“行为治理”问题。

### 第 1 章 为什么 Harness Engineering 不是 Prompt Engineering 的大号别名

这一章是在拆误解。很多人会把 agent 系统理解成“更复杂的 prompt 技巧”，但作者强调，真正决定系统质量的，是 prompt 之外那一整套运行时结构。这里是整本书的地基。

### 第 2 章 Prompt 不是输入框，而是控制面

这一章把 prompt 的角色重新定义了。它不只是模型看到的一段文字，而是系统对行为边界、优先级和执行风格的持续约束。换句话说，prompt 在这里承担了一部分控制面职责。

### 第 3 章 Query Loop：Agent 不是在答题，而是在持续接管工作流

这一章是本书的心脏。Claude Code 之所以像一个 agent，而不是一个聊天机器人，是因为它围绕 `query loop` 反复读取状态、调用工具、更新上下文、决定下一步动作。理解这一章，基本就理解了“为什么它不是一次性回答系统”。

### 第 4 章 工具、权限与中断：怎样让模型动手，但不让它乱动手

这一章关注的是工具调用和权限控制。模型要有行动能力，但行动能力必须被沙箱、审批、白名单、边界检查和中断机制组织起来。这里讲的是“放权”和“刹车”怎样共存。

### 第 5 章 上下文、记忆与压缩：怎样让系统长期工作而不是越聊越糊

长任务里最大的问题不是一次答错，而是系统越做越失焦。这一章讨论上下文窗口、摘要、压缩、持久记忆和状态重建，核心目标是让系统在长周期里仍然保留方向感。

### 第 6 章 错误与恢复：模型犯错不是异常，而是运行时常态

这章很关键，因为它把“出错”从事故改成常态。模型会误判、会误操作、会偏题，所以系统必须有恢复路径、回滚思路、重试机制和状态修复能力。成熟系统不是不犯错，而是犯错后还能回到正轨。

### 第 7 章 多 Agent 与验证：不要让系统自己给自己当裁判

这一章提醒一个常见误区: 多 agent 不该只是“多开几个模型实例”，而应该明确谁执行、谁评审、谁验证、谁做独立检查。系统要避免把生成和验收混成同一条神经回路。

### 第 8 章 团队落地：把个人技巧变成组织能力

Claude Code 这类系统真要在团队里跑起来，不能靠高手的临场发挥。这一章关注的是技能沉淀、流程约束、本地规则、可审计性和组织学习。它把问题从“个人生产力”推到“团队制度设计”。

### 第 9 章 十条原则：如何判断一个 AI 编程系统是不是工程系统

这是全书最适合反复回看的部分。前面几章提供推理链条，这一章把结论压缩成判断标准。你可以把它当成选型清单，也可以拿来审视自己做的 agent 系统有没有工程骨架。

### 附录 A 检查清单

适合直接拿来做自检。无论是评估第三方工具，还是检查自己团队内部 agent 的成熟度，这部分都很实用。

### 附录 B 图稿说明

适合在你已经读完整本书后回看，用来把抽象结构和图示重新对齐。

### 附录 C 源码地图

如果你想把书里的判断往实现层落，这一部分非常重要。它把概念和代码位置对起来，方便从“抽象结构”切回“真实实现”。

## 建议读法

1. 如果你是第一次系统理解 Claude Code，建议按顺序读到第 4 章。
2. 如果你主要关心长期运行问题，重点看第 5 到第 7 章。
3. 如果你是团队负责人或平台负责人，第 8 章和第 9 章优先级最高。
4. 如果你已经看过源码或相关解析，附录 C 很值得对照着读。
