## 先给一个判断

这本书最有价值的地方，不是给 `Claude Code` 或 `Codex` 做胜负判定，而是把两种 `harness` 设计思路并排摆出来，让你看清楚: 同样是在做 AI 编程系统，秩序到底被安放在哪一层。

它比的不是功能表，而是这些更底层的问题：

- 控制面怎样组织
- 主循环和线程状态怎样表达
- 权限、沙箱和策略语言怎样配合
- 技能、Hook 和本地规则怎样把组织习惯写进系统
- 委派、验证和持久状态怎样防止系统自我欺骗

如果第一本书是“把 Claude Code 的骨架讲清楚”，第二本书就是“把两种骨架的分歧讲清楚”。

## 适合谁读

1. 已经用过 `Claude Code` 或 `Codex`，想看设计分歧的人
2. 正在做 AI coding agent 选型或平台架构判断的人
3. 想从具体产品抽象出通用 harness 原则的人
4. 希望理解 `query loop`、`thread`、`rollout`、`state` 等结构职责边界的人

## 这本书最值得抓住的 6 个问题

1. Claude Code 和 Codex 的控制面分别压在什么层
2. 两套系统的运行时心跳是怎样组织的
3. 权限和策略是更像运行时规训，还是结构化治理
4. 本地规则、技能和 Hook 在不同系统里扮演什么角色
5. 多 agent、验证和持久状态怎样避免“自己给自己打高分”
6. 如果你要自己做一套 harness，应该优先学谁的哪一层

## 章节地图

### 阅读地图：如何理解第一本书与这本比较书

这部分的作用是重新校准读者位置。它提醒你，这不是一本孤立的比较书，而是建立在第一本书已经讲清控制骨架之后，进一步比较不同实现哲学。

### 序言：两套 Harness，不必假装是同一匹马的附件

这部分很重要，因为它避免了“强行对齐”。作者的态度不是把两个系统揉成同一套模板，而是承认它们各自有不同的秩序安放方式。

### 第 1 章 为什么要把 Claude Code 和 Codex 放在一起看

这章在回答“比较有没有意义”。答案是有，因为只有在并排对照时，很多原本习以为常的设计选择才会暴露出来。

### 第 2 章 两种控制面：Prompt 拼装与 Instruction Fragment

这是整本书的第一条主线。控制面到底是更靠近 prompt 组织，还是更靠近结构化 instruction 片段，是两套系统的关键分歧之一。这里读懂了，后面很多差异都会更容易对齐。

### 第 3 章 心跳放在哪：Query Loop 对照 Thread、Rollout 与 State

这章讨论系统的“持续性”被放在哪里。Claude Code 更强调运行时循环的连续接管，而 Codex 系列更容易让你从线程、rollout、状态机这些结构化抽象去理解执行过程。

### 第 4 章 工具、沙箱与策略语言：谁来阻止模型动手太快

这章关注权力怎么被分配。工具是能力入口，沙箱是边界，策略语言是显式治理层。系统如何组合这三者，直接决定了执行自由度和风险控制方式。

### 第 5 章 技能、Hook 与本地规则：系统如何学会守乡约

这一章我很推荐细读。因为真正让 agent 在组织里稳定工作的，往往不是基础模型能力，而是团队把习惯、约束和本地制度写进系统的方式。这里是在讨论“组织经验怎么被编码”。

### 第 6 章 委派、验证与持久状态：谁来防止系统自己给自己打高分

这章讨论的是可信性。一个能委派的系统，不等于一个能自证正确的系统。要防止生成、评估和执行互相串通，就要重新划分验证与状态职责。

### 第 7 章 殊途同归，还是各表一枝

这是全书的阶段性结论。作者在这里会把前面的分歧重新收拢，回答两个问题: 这些系统究竟是在趋同，还是在形成不同门派；这些差异又会怎样影响未来产品形态。

### 第 8 章 如果你要自己做：该向谁学，先学什么

这是最适合工程团队和创业团队的一章。它不是停留在比较，而是逼你做选择: 如果资源有限，哪一层该先学，哪一层该后学，哪些结构先搭最划算。

### 附录 A 源码地图

适合把比较结果和实现证据对起来看。你如果不想只停留在抽象判断，这部分很值得保留。

### 附录 B 检查清单

这部分适合实战。尤其是当你开始评估自己的系统更像哪一路，或者仍停留在“半成品”阶段时，它会很有帮助。

## 推荐读法

1. 最好先读 [Harness Engineering：Claude Code 设计指南](/sources/harness-book-claude-code-guide)，再回来读这本比较书。
2. 如果你已经很熟悉 Claude Code 或 Codex，可以从第 2 章到第 6 章直接切入。
3. 如果你关心选型和路线判断，第 7 章和第 8 章最关键。
4. 如果你准备自己做系统，附录 B 可以作为设计评审清单。

## 这本书对 `aithink` 的价值

`aithink` 不只是要收集“最好用的 AI 工具”，还要尽量解释清楚: 为什么有些工具能长期工作，而有些只能演示。第二本书的价值，就在于它把“产品差异”翻译成“控制结构差异”。

这会直接帮助我们判断：

- 一个系统的秩序是写在 prompt 里，还是写在 runtime 结构里
- 一个系统的安全边界主要靠人盯，还是靠机制分层
- 一个系统的团队经验是否能沉淀成稳定制度

## 推荐搭配阅读
