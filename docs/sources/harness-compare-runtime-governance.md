# 精读 5：运行时治理，主循环、沙箱、技能与验证如何协同

> 这篇精读主要对应《Claude Code 和 Codex 的 Harness 设计哲学》的中段内容，聚焦主循环、状态表达、沙箱、技能、本地规则和验证机制怎样共同形成运行时治理。

原始来源：

1. [在线阅读](https://harness-books.agentway.dev/book2-comparing/)
2. [GitHub 仓库](https://github.com/wquguru/harness-books)

## 先说结论

真正决定一个 AI coding system 是否可靠的，不是它某个局部功能有多强，而是它的运行时治理是否成体系。

这本比较书中段最精彩的地方，就是把这些原本容易分散看的部件重新绑在一起：

- 主循环或线程状态
- 工具调用
- 沙箱边界
- 策略语言
- 技能和 Hook
- 本地规则
- 验证机制

作者的意思很明确：这些不是外围装饰，而是同一套治理系统的不同器官。

## 主循环和线程状态到底在解决什么

无论你叫它 `query loop`、`thread`、`rollout` 还是 `state`，本质都在回答同一个问题：

系统怎样记住自己正在做什么，并把下一步动作接上去。

这一步如果做不好，系统就会出现几种典型问题：

- 任务中途失焦
- 动作和状态脱节
- 工具结果无法稳定并回上下文
- 长任务里越来越依赖临场猜测

所以运行时心跳并不是产品细节，而是秩序的载体。

## 沙箱和策略语言为什么是治理层，不只是安全层

很多人第一次看到沙箱，会把它理解成“安全功能”。这当然没错，但还不够。

更准确地说，沙箱和策略语言是在表达系统允许怎样使用权力。

它们回答的不是“能不能执行命令”这么简单，而是：

- 在什么条件下可以执行
- 哪些路径可读写
- 什么时候需要审批
- 失败后如何限制影响范围
- 哪些高风险动作必须被显式阻断

这就让沙箱从一个防护罩，升级成了一层治理结构。

## 技能、Hook 和本地规则，为什么是组织经验的编码方式

这部分是我很喜欢的段落。因为它把“团队经验”从口头技巧升级成了系统设计对象。

任何一个真实团队，最终都会长出自己的“乡约”：

- 哪些命令要谨慎
- 哪些目录不要动
- 哪些改动前先看测试
- 哪些情况下必须先问人

如果这些经验不能被写进系统，它们就只能依赖高手记忆。  
一旦高手不在，系统秩序就会明显退化。

所以技能、Hook 和本地规则的价值不只是扩展能力，而是把组织习惯写成可复用结构。

## 为什么验证一定要独立出来

这一部分和第一本书是呼应的。  
当系统越来越会委派、越来越能自动推进任务时，验证机制就不再是锦上添花，而是防止系统自我陶醉的必要结构。

如果一个系统可以自己生成、自己解释、自己打分，那它在演示里会非常顺，但在真实工程里往往风险很高。

所以比较一个 agent 系统是否成熟，你可以直接看：

1. 生成和验证是否角色分离
2. 状态和结论是否可追溯
3. 高风险动作是否有额外检查层

## 我对这一段内容的点评

这段比较最有价值的地方，是它把“主循环”“沙箱”“技能”“验证”这些看似离散的话题重新并进为一个判断框架。

很多产品介绍会把这些点拆开讲，于是读者会误以为它们是功能菜单。  
但这本书提醒我们，这些组件之间其实有很强的依赖关系：

- 没有稳定状态表达，验证就会失去依据
- 没有沙箱边界，工具能力越强风险越大
- 没有本地规则，团队经验就很难沉淀
- 没有独立验证，系统很容易在自我解释里打转

## 这篇读完后应该带走什么

1. 运行时治理不是一个功能点，而是一组互相咬合的结构。
2. 沙箱和策略语言不是附属安全层，而是权力分配方式。
3. 技能、Hook 和本地规则决定了系统能否吸收组织经验。
4. 验证机制是成熟 agent 的必要器官，不是可选外挂。
