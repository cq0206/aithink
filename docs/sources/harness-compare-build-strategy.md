# 精读 6：如何选型与自建，从比较走向工程决策

> 这篇精读主要对应《Claude Code 和 Codex 的 Harness 设计哲学》的后半部分。重点不是继续做差异罗列，而是把这些比较压缩成选型标准和自建顺序。

原始来源：

1. [在线阅读](https://harness-books.agentway.dev/book2-comparing/)
2. [GitHub 仓库](https://github.com/wquguru/harness-books)

## 先说结论

比较 `Claude Code` 和 `Codex` 的真正价值，不在于选一个“更强”的，而在于逼你回答：如果你自己做系统，你到底要先学什么、先搭什么、把秩序放在哪一层。

这篇的重点可以压成三个问题：

1. 你的场景更需要流畅的一体化体验，还是更需要显式治理能力
2. 你的团队现在最缺的是运行时纪律，还是结构化控制层
3. 你是要选工具，还是要学会设计自己的 harness

## 如果你在做工具选型，该怎么想

我的建议不是看“谁功能更多”，而是先看下面这几项：

### 先看秩序安放在哪里

你要判断的是：

- 这个系统的约束主要写在哪一层
- 用户是否容易感知这些约束
- 团队规则能否稳定注入

如果秩序安放位置模糊，系统短期可能很好玩，但长期很难治理。

### 再看运行时纪律够不够强

一个系统能不能长期工作，往往取决于这些问题：

- 长任务里会不会失焦
- 工具调用是否有稳定边界
- 失败后是否有恢复路径
- 验证机制是否独立存在

这些东西往往比“生成效果偶尔惊艳”更决定长期价值。

### 最后看它能不能进入团队制度

真正能落地的工具，一定允许团队把经验写进去。  
否则它只能是高手专用效率外挂。

所以你应该继续问：

- 能不能写本地规则
- 能不能做权限治理
- 能不能形成可追溯的工作方式
- 能不能让新人继承成熟经验

## 如果你要自己做一套 harness，先搭什么

这本书最后一章最有价值的地方，是把“学习顺序”也摆了出来。  
我很认同这种思路，因为很多团队一上来就想做大而全，结果反而把最关键的骨架漏掉。

如果让我把顺序压成一个更实际的版本，我会建议：

1. 先搭控制面
2. 再搭主循环和状态表达
3. 再接工具和权限边界
4. 再做恢复和验证
5. 最后才是多 agent 扩展和团队制度化

这个顺序背后的逻辑很朴素：

- 没控制面，就没有秩序
- 没主循环，就没有持续性
- 没权限边界，行动能力会直接变风险
- 没恢复和验证，系统很难被信任
- 没制度化，团队规模一上来就会散

## 我对这本比较书最终结论的理解

我不觉得它最后是在宣布某一方胜出。它更像是在说：

- 两套系统都在回答同一类工程问题
- 但它们安放秩序的方式不同
- 所以适合学习和借鉴的层次也不同

这会逼读者进入一个更成熟的视角：

不是问“哪个更厉害”，而是问“我的问题更像哪一种，因此我该先学哪一种骨架”。

## 一份可直接拿去用的选型清单

如果你准备评估 AI coding agent，我建议至少看这 8 项：

1. 控制面是否清晰
2. 主循环或状态表达是否稳定
3. 工具权限是否可治理
4. 沙箱边界是否明确
5. 本地规则和组织习惯能否注入
6. 验证是否独立于生成
7. 出错后的恢复路径是否明确
8. 团队能否把经验沉淀成制度

如果这 8 项里有 4 项都说不清，那系统大概率还停留在演示级。

## 我对这篇的点评

我很喜欢这本书最后的落点，因为它没有停在“分析得很漂亮”。它逼着你把分析变成行动顺序。

这对 `aithink` 也很重要。  
我们不是为了堆概念，而是为了筛出那些真正能帮助你做判断、做选型、做系统设计的内容。

所以如果你读完这本比较书，脑子里只留下“Claude Code 和 Codex 不一样”，那还不够。  
更重要的是，你要能回答：

- 我的团队现在缺什么
- 我的系统秩序该安放在哪里
- 我应该先学哪一层，再学哪一层
