# 精读 1：为什么 Prompt 不够，控制面才是 Claude Code 的骨架

> 这篇精读基于《Harness Engineering：Claude Code 设计指南》的前两章重写，目标不是复述原文，而是把其中最重要的判断变成适合网页阅读的一篇文章。

原始来源：

1. [在线阅读](https://harness-books.agentway.dev/book1-claude-code/)
2. [GitHub 仓库](https://github.com/wquguru/harness-books)

## 先说结论

很多人第一次理解 `Claude Code`，会把它当成一个“提示词很强的 coding CLI”。这个看法不算错，但太浅。因为真正让它形成工程能力的，不只是 prompt，而是 prompt 背后那整套控制面。

这篇要抓住的核心判断只有一句话：

`Claude Code` 之所以像一个系统，不是因为它会回答，而是因为它被一层持续存在的控制结构约束着行动。

## 为什么 Prompt 不够

如果系统只是“用户输入一句话，模型生成一段回答”，那 prompt 确实几乎就是全部世界。但 Claude Code 不是这样工作的。

它面对的是另一类任务：

- 读取项目上下文
- 分析文件和状态
- 规划接下来要做什么
- 调用工具去执行动作
- 根据执行结果再决定下一步

一旦任务从“回答问题”变成“接管工作流”，prompt 就不可能独自承担全部治理责任。因为它很难同时稳定处理：

1. 行为边界
2. 优先级冲突
3. 权限控制
4. 长任务一致性
5. 错误恢复
6. 团队制度沉淀

这也是这本书强调 `Harness Engineering` 的原因。它要解决的不是“怎样让模型更会说”，而是“怎样让模型进入真实工程环境后仍保持秩序”。

## Prompt 在这里到底是什么

这本书对 prompt 的重新定义很值得记住。

在普通聊天产品里，prompt 更像一次输入。
但在 Claude Code 这类系统里，prompt 更像控制面的一部分。

它承担的职责包括：

- 给模型设定总目标和角色
- 指定行为风格和优先级
- 写入边界、禁令和默认策略
- 把局部任务放进更大的工程上下文

所以你看到的并不是“一段提示词”，而是一块被嵌入运行时的治理板。

## 什么叫控制面

如果要用更工程一点的话来说，控制面就是那层不直接做业务动作，但持续决定系统怎样做动作的结构。

它不等于某一个单独文件，也不等于某一段提示词。它更像这些东西的组合：

- 系统级指令
- 工具权限规则
- 任务优先级
- 失败后的默认处理方式
- 用户确认与审批机制
- 本地项目规则

这些东西合起来，才形成了一个 agent 可以被托付工作的基础。

没有控制面，模型再聪明，也只是一个临场发挥的执行器。  
有了控制面，它才开始像一个能被纳入工程秩序的系统。

## 为什么这件事对 Claude Code 特别重要

Claude Code 的特殊点在于，它不只是生成代码，它还会进入仓库、调用命令、修改文件、读写状态。也就是说，它的输出不是一段文字，而是会留下后果的动作。

这会让系统进入一个完全不同的风险等级：

- 回答错了，最多是信息偏差
- 改错文件，可能直接造成回归
- 误用命令，可能污染工作区
- 读错上下文，会让后续整条执行链偏航

在这种环境里，prompt 当然重要，但 prompt 已经不是主角了。主角变成了“系统怎样组织行为后果”。

## 这篇精读里最重要的 3 个转向

### 从“回答质量”转向“行为后果”

你评价一个 agent，不该只问它写得像不像，还要问它的动作有没有边界、有没有恢复能力、有没有可审计性。

### 从“提示词技巧”转向“运行时结构”

让模型更会说，并不等于让系统更可靠。真正让系统稳定的，是控制面、主循环、权限、记忆和验证这些结构。

### 从“个人提效工具”转向“工程系统”

如果一个东西只能在高手手里工作，它就还不是工程系统。工程系统的标准，是换一批人、换一个项目、换一段时间后，它仍然有秩序。

## 我对这两章的点评

这两章最强的地方，是把很多人直觉里“好像哪里不对”的感觉说清楚了。  
大家都知道只堆 prompt 不够，但到底不够在哪，往往说不完整。这本书把缺失的那一层命名成 `Harness Engineering`，于是问题突然就立住了。

它不是在否认 prompt 的价值，而是在帮读者升级问题定义：

- prompt 很重要
- 但 prompt 不是全部
- 真正决定系统能不能长期工作的是控制结构

这个转向一旦成立，后面关于 `query loop`、权限、恢复和团队制度的章节就都会变得顺理成章。
