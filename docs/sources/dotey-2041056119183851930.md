# 宝玉：Codex 团队如何用产品构建产品（2026-04-06）

## 点评

这条内容价值很高，因为它不是“功能介绍”，而是给出了 Codex 团队真实的组织与产品运作信号：

1. `Spec` 极度压缩：不是每个功能 10 条，而是整个产品 spec 也可以只有 10 条要点。
2. 规划反常识：只做近端（8 周内）和远端（方向感），刻意不做中期路线图。
3. 角色边界重排：PM、设计、工程之间的传统分工正在被 Agent 工作流重塑。
4. 产能约束转移：重点从“写代码”转到“选对问题 + 质量保障 + 系统性验证”。
5. 社区前沿共创：通过开源和高能用户反馈，产品方向由“真实使用”持续拉动。

如果你在建设 agent-first 团队，这条内容的启发不是“照抄组织结构”，而是识别它成立的前提：
- 团队本身就是目标用户
- 模型能力持续跃迁
- 工具链可观测且反馈闭环足够快

## 可执行提炼（给团队）

1. 把“中期路线图”改成“8 周目标 + 年度方向假设”，每周重估。
2. 用 `plan mode` 先探索代码库可行路径，再写最小决策文档。
3. 让“离代码最近的人”拥有更多决策权，同时把架构约束写进工具链。
4. 默认要求每次改动附带质量证据（测试、回归验证、监控影响面）。
5. 对外建立高频反馈回路，把重度用户行为当作产品方向信号。
