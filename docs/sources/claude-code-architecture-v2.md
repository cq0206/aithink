# Claude Code 源码架构深度解析 V2.0

> 说明：本文基于用户提供的 PDF [ai-agent-deep-dive-v2.pdf](/Users/qian/Documents/图片/03/ai-agent-deep-dive-v2.pdf) 整理成网页阅读版，适合放进 `aithink` 做长期查阅。

## 这篇内容在讲什么

这份 PDF 的核心价值，不是“Claude Code 有多少文件”，而是它把一个成熟 Agent 系统拆成了可以复用的工程设计模式。

如果你把 Claude Code 只看成一个 CLI 工具，很多设计会显得过度复杂；但如果把它看成一个 Agent Operating System，这套架构就立刻合理了。它的真正目标不是回答一次问题，而是支撑一个长期运行、可扩展、可治理、可验证、可产品化的智能体系统。

## 一句话结论

Claude Code 的源码传递出的主线非常明确：

1. 它不是“模型 + 几个工具”的轻量封装，而是一套平台化的 Agent 运行时。
2. 它最强的部分不是模型本身，而是主循环、工具治理、多 Agent 分工、安全层和上下文经济学。
3. 它真正做对的地方，是把“模型不可靠”当成前提，然后在运行时里用制度、权限、角色拆分和缓存策略去兜底。

## 全局视角：它不是 CLI，而是 Agent OS

这份报告先从代码库规模入手。作者从 `cli.js.map` 的 `sourcesContent` 中提取出完整 TypeScript 源码，统计出接近 `4756` 个文件。这个数字本身就说明 Claude Code 的重点从来不是“小而美的命令行工具”，而是一个平台级系统。

从目录分布也能看出这一点：

- `utils/` 体量最大，说明底层能力和系统粘合层非常厚
- `components/` 和 `ink/` 很大，说明终端 UI 不是附属品，而是产品界面的一部分
- `commands/`、`tools/`、`services/`、`hooks/` 都是独立大块，说明它在命令面、工具面、服务面和治理面分别做了平台化设计

它还有多个入口：`cli.tsx`、`init.ts`、`mcp.ts`、`sdk/`。这意味着同一套 agent runtime 被复用于 CLI、MCP、SDK 和 IDE 集成，而不是每个渠道各造一套。

一个很典型的产品化细节是 CLI 的 fast-path：

- `--version` 不加载完整运行时
- `remote-control` 直接进入 bridge
- `daemon` 走长驻进程
- `ps / logs / attach / kill` 走后台会话管理
- `--worktree --tmux` 直接 `exec` 进 tmux

只有这些都没命中，才真正加载主应用。这不是炫技，而是典型的平台思维：高频、短路径、无需完整上下文的调用，必须用最短链路完成。

## 引擎层：真正的核心是主循环和 Prompt 编排

这份 PDF 里最重要的判断之一，是 `query.ts` 才是整个系统的心脏。

从调用链来看，一次请求大致会这样跑：

1. CLI 分发到 `main.tsx`
2. 初始化状态和工具上下文
3. 用户输入进入 `query()`
4. `query()` 进入 `queryLoop()` 的 `while(true)` 主循环
5. 每轮迭代完成：上下文压缩、prompt 组装、模型调用、工具执行、结果注入、继续或退出

这里最值得学的不是“循环”这个形式，而是它把 agent 行为写成了一个显式状态机。相比递归调用，`while(true) + state` 的设计更适合长会话，因为：

- 不会爆栈
- 状态切换更清晰
- 每一种“为什么继续下一轮”的理由都能变成独立分支
- 错误恢复、重试、budget 控制都更容易制度化

作者还特别点出一个优化：Streaming Tool Execution。传统做法是等模型把整段输出都吐完，再统一执行所有 `tool_use`。Claude Code 选择边收边跑，已经收完整的工具调用就立即执行。这会直接改善交互体验，尤其在一轮里包含多个工具调用时，延迟会明显降低。

另一块工程味非常重的设计是 Prompt 编排。`prompts.ts` 不是简单拼一段 system prompt，而是把静态部分和动态部分明确拆开，中间用 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 分界：

- 静态部分：身份、系统规范、任务规范、动作规范、语气风格、输出效率
- 动态部分：session guidance、memory、环境信息、语言偏好、MCP instructions、token budget 等

这里最值得注意的是 cache 设计。系统显然在为 prompt cache 命中率做工程优化：不变的内容尽量保持字节级一致，变化的内容统一放在后面。这样多轮请求和子 agent fork 都能吃到更多缓存命中，降低成本和延迟。

## Prompt 里最值钱的部分：行为制度化

报告对 `getSimpleDoingTasksSection()` 的评价非常高，我同意这一点。

因为成熟的 Agent 系统，关键从来不是“让模型更聪明”，而是“让模型少犯那些一再重复的错”。这部分 prompt 明确写了很多 coding agent 常见反模式：

- 不要加用户没要的功能
- 不要过度抽象
- 不要给没改过的代码乱加注释
- 不要无意义地做兜底逻辑
- 不要设计面向未来的抽象
- 先读代码，再改代码
- 不要轻易新建文件
- 没跑过的不要说跑过了

这些规则的价值在于，它们不是依赖模型“自己悟到”，而是把行为预期制度化。换句话说，Claude Code 的设计哲学不是“相信模型会自觉”，而是“默认模型会犯错，所以先写规则”。

## 工具系统：真正成熟的地方在治理，而不是数量

PDF 提到 Claude Code 一共有 `42` 个工具。这个数字本身不算最重要，真正关键的是工具执行 pipeline 的严密程度。

一条完整的工具调用链不是“模型说调就调”，而是至少要经过这些环节：

1. 根据名字找到工具
2. 解析 MCP 元数据
3. Zod schema 校验
4. 工具自身的细粒度校验
5. 风险预判分类器
6. PreToolUse hooks
7. Hook 返回的权限决策解析
8. 最终权限决策
9. 输入修正
10. 真正执行
11. analytics / tracing / OTel
12. Post hooks
13. 结果结构化处理
14. 失败 hook

这套设计最成熟的地方在两个词：`fail-closed` 和 `分层治理`。

比如新工具如果忘了声明是否并发安全，系统默认它不安全；忘了声明是否只读，系统默认它可能会写。这种“忘了就更严格”的默认值，能显著减少配置遗漏导致的安全事故。

再比如 BashTool 的 speculative classifier，会在权限决策前并行启动，对命令风险做预测。它不是最终裁决者，但它能让风险判断提前完成，从而改善权限交互延迟。

## 多 Agent：不是多开几个模型，而是拆角色

Claude Code 至少有 6 个内建 Agent。报告里最值得关注的不是“数量”，而是角色切分方式：

- General Purpose Agent：通用任务执行
- Explore Agent：只读探索
- Plan Agent：只做规划
- Verification Agent：对抗性验证
- Claude Code Guide Agent：使用指导
- Statusline Setup Agent：状态栏配置

这里最成熟的做法，是把“实现者”和“验证者”拆开。Verification Agent 的设计非常狠，它的核心目标只有一个：`try to break it`。

它甚至明确点出两种典型验证失败模式：

- Verification avoidance：只看代码，不真正跑验证
- 被前 80% 迷惑：界面看着还行、测试也过了，就默认剩下 20% 没问题

这和传统软件工程里的分离验收职责是同一种思想。写代码的人天然更容易相信自己没写错；独立验证角色的存在，就是为了专门对抗这种偏差。

Explore Agent 也很典型：它被彻底限制为只读，不允许新建文件、不允许修改文件、不允许做任何改变系统状态的动作。这种权限裁剪虽然朴素，但非常有效，因为它直接阻断了“探索阶段误改代码”的风险。

## 安全层：真正成熟的是“互不绕过”

报告把 Claude Code 的安全层总结为三层防护网：

1. Speculative Classifier：提前预判风险
2. Hook Policy Layer：运行时策略层
3. Permission Decision：最终权限裁决

最关键的一点是：这三层可以互相配合，但不能互相绕过。

例如：

- Hook 返回 `allow`，并不意味着可以绕过 settings 里的 `deny`
- Hook 说 `ask`，仍然可能进入权限弹窗
- Bash 风险分类只是辅助信息，不能替代最终权限决策

这类设计特别像成熟安全系统的思路：允许局部策略参与，但必须保留核心裁决层的最高优先级。这样即使某个 hook 写坏了，或者被恶意利用，也不至于直接击穿整套安全模型。

## 生态层：Skill、Plugin、MCP 的关键不是“有”，而是“模型知道自己有”

PDF 里对生态系统的总结非常准确：很多平台也有插件、也有工具市场，但模型本身根本不知道这些扩展存在，于是等于没接。

Claude Code 这里真正做对的是“能力感知”：

- Skill 按任务匹配注入
- Plugin 可以影响 prompt、工具、权限规则和输出风格
- MCP 不只是注册新工具，还能把 `instructions` 注入 system prompt
- Session-specific guidance 会告诉模型当前到底有哪些能力可以用

这意味着模型不只是“可以调用某个能力”，而是“知道什么时候该调用、为什么该调用、现在能不能调用”。这一步才是生态从“扩展清单”变成“真实能力网络”的关键。

## 上下文经济学：把 token 当预算，而不是当空气

这份 PDF 还有一个我很认同的判断：上下文管理本质上是预算管理。

Claude Code 在这里的设计很完整，至少包括：

- Snip Compact
- Micro Compact
- Context Collapse
- Auto Compact
- Reactive Compact
- Token Budget
- 按需注入 skills 和 MCP instructions
- 预取 memory 与 skill discovery
- 大工具结果落盘，只把摘要放回上下文

这背后反映的是一种很成熟的认知：每个 token 都有成本，每条上下文都占空间，能缓存的缓存，能延迟注入的延迟注入，能压缩的尽量压缩。

很多 agent 系统做不长，不是因为模型不够强，而是因为上下文管理太粗糙。Claude Code 在这方面花了很多系统工程力气。

## 产品化：真正的难点在“第二天”

PDF 里有一句话特别好：大多数 Agent 系统第一天都能跑起来，问题出在第二天。

这句话点到了产品化的真正难点：

- 任务中断怎么续
- 子 agent transcript 怎么记
- 脏状态怎么清
- shell 进程怎么杀
- MCP 连接泄漏怎么办
- session hooks 怎么清理
- todos、tracking、perfetto tracing 怎么收尾

这类问题不会让 demo 立刻失败，但一定会让产品迟早出问题。所以成熟系统和原型系统的分水岭，往往不在“首轮体验”，而在生命周期管理。

Claude Code 在 `runAgent.ts`、bridge 系统、全局 state 管理、终端 UI 和 telemetry 里，明显已经进入了“产品运行时”思维，而不是“研究原型”思维。

## 从源码里最值得带走的 7 条设计原则

结合 PDF 最后几页，我觉得这 7 条可以直接当成构建 Agent 系统时的检查清单：

1. 不要信任模型的自觉性，要把好行为写成制度。
2. 把探索、规划、实现、验证拆成不同角色。
3. 工具调用必须有治理链路，而不是直连执行。
4. 上下文是预算，缓存、压缩、按需加载都要做。
5. 安全层之间必须互不绕过。
6. 生态的关键不是插件数量，而是模型是否感知自己的能力。
7. 产品化不在于 demo 跑通，而在于第二天还能继续稳定运行。

## 对我们自己做 Agent 系统的启发

如果把这份 PDF 浓缩成对工程实践最有价值的几句话，我会总结成下面这些：

- 不要把 Agent 当成“模型 + 工具调用”，而要把它当成一个运行时系统。
- 多 Agent 不是为了显得高级，而是为了拆分认知职责和降低偏差。
- Prompt 工程真正成熟的标志，不是写得多漂亮，而是能否把错误模式制度化地压下去。
- 权限、安全、hook、风险分类必须是协同设计，而不是各管一摊。
- 真正决定产品体验的，往往不是模型本身，而是主循环、上下文压缩、工具执行时机和生命周期管理。
