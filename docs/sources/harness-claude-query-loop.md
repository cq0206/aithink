# 精读 2：Query Loop、工具权限与中断，Claude Code 如何持续接管工作流

> 这篇精读主要对应《Harness Engineering：Claude Code 设计指南》的第 3 章和第 4 章，重点看 Claude Code 为什么不是一次性答题系统，而是一套持续运转的工作流接管机制。

原始来源：

1. [在线阅读](https://harness-books.agentway.dev/book1-claude-code/)
2. [GitHub 仓库](https://github.com/wquguru/harness-books)

## 先说结论

`Claude Code` 的关键不是“会不会写代码”，而是它能不能在一连串动作里持续保持方向、边界和节奏。支撑这一点的核心结构，就是 `query loop`。

如果说控制面决定“应该怎样工作”，那么 `query loop` 决定“工作怎样持续发生”。

## 什么是 Query Loop

最简单的理解方式是，把它看成一颗持续跳动的心脏。

它反复做几件事：

1. 读取当前任务和环境状态
2. 判断下一步该做什么
3. 选择是否调用工具
4. 处理工具返回的结果
5. 更新上下文和局部记忆
6. 再进入下一轮判断

这跟一次性问答完全不同。一次性问答只关心“现在回答什么”，而 `query loop` 关心“现在做什么，以及做完之后怎样继续做”。

## 为什么说 Agent 是在接管工作流

这本书有一个很重要的提醒：agent 不是在参加考试，它是在接手一段工作。

工作有几个特征：

- 状态会变化
- 结果会反过来影响后续步骤
- 中途会遇到失败和分支
- 工具调用的后果是真实存在的

所以 Claude Code 的价值，不只是能提出答案，而是能在多轮动作中持续对齐目标。

## 工具为什么是能力入口，也是风险入口

一旦模型能调用工具，系统就从“能说”升级成了“能动手”。这当然会大幅提升能力，但风险也会同步放大。

常见风险包括：

- 误改文件
- 跑错命令
- 读取不该读取的路径
- 在错误上下文上继续推进任务

所以工具系统必须和权限系统一起设计。  
书里这一段非常重要，因为它把“会调用工具”从功能点重新定义成了治理问题。

## 权限和中断为什么要一起看

很多人会把权限理解成静态白名单，把中断理解成异常处理。但在 agent 系统里，这两者其实是同一个问题的两面：

- 权限是在动作发生前定义边界
- 中断是在动作进行中重新夺回控制权

两者一起构成“放权但不失控”的基础。

如果没有中断，系统一旦判断失误，就会把错误一路放大。  
如果没有权限，系统虽然可以被中断，但每次失误都可能已经留下很大后果。

## 这两章真正要解决什么

我觉得这两章其实在回答一个很现实的问题：

一个模型如果已经足够聪明，会不会就自然变成好用的 agent？

作者给出的答案很明确：不会。

因为好用的 agent 不是由单次推理能力决定的，而是由这些东西共同决定的：

- 有没有稳定的执行心跳
- 有没有可控的工具入口
- 有没有分层的权限边界
- 有没有在关键时刻人工接管的办法

也就是说，运行时的秩序比单步推理更重要。

## 我对这两章的点评

我很认同这两章的重心安排。很多关于 AI coding agent 的讨论，容易把注意力放在“生成结果像不像资深工程师”，但真正进入工程环境后，最先暴露问题的通常不是智力，而是节奏和边界。

`query loop` 解决的是节奏问题。  
权限和中断解决的是边界问题。

两者合起来，Claude Code 才从一个有能力的模型，变成一个可以被慢慢信任的执行系统。

## 如果你要自己做类似系统，这里有 4 个启发

1. 不要只设计 prompt，要设计循环。
2. 不要只设计工具，要设计权限。
3. 不要只设计成功路径，要设计中断路径。
4. 不要只看单步效果，要看多步任务中的偏航积累。
