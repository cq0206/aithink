# 精读 4：两种控制面，Claude Code 与 Codex 到底从哪里安放秩序

> 这篇精读主要对应《Claude Code 和 Codex 的 Harness 设计哲学》的前半部分，尤其是“为什么比较”和“控制面差异”这两条主线。

原始来源：

1. [在线阅读](https://harness-books.agentway.dev/book2-comparing/)
2. [GitHub 仓库](https://github.com/wquguru/harness-books)

## 先说结论

比较 `Claude Code` 和 `Codex`，最重要的不是看谁功能更多，而是看这两套系统把秩序安放在什么地方。

这是一个非常工程的问题。因为秩序放在哪一层，会直接决定：

- 系统如何表达约束
- 用户怎样理解边界
- 团队怎样写入自己的规则
- 运行时如何处理风险和分工

## 为什么这两者值得放在一起看

这本书的一个亮点，是它没有把比较做成功能评测，而是做成了架构显影。

平时你单独看一个系统，很多设计选择会显得理所当然。  
但一旦把两套系统并排放在一起，很多“默认值”就会突然暴露出来。

比如：

- 为什么有的系统更强调 prompt 层组织
- 为什么有的系统更强调结构化 instruction 片段
- 为什么有的系统让你先感受到运行时纪律
- 为什么有的系统让你先感受到显式的控制层

这种对照，本身就是理解系统的最好方法。

## 两种控制面有什么根本差别

这本书提到的关键差异之一，是控制面表达方式不同。

我的理解可以简化成这样：

- `Claude Code` 更容易让人感受到一套围绕运行时展开的控制面
- `Codex` 更容易让人感受到一套结构化、更显式可拆分的控制层

这不是说谁更高级，而是说两者回答“约束应该被写在哪里”这个问题时，重心不同。

这种不同会影响很多下游设计：

- 规则组合方式
- 本地习惯注入方式
- 人对系统边界的感知方式
- 系统如何扩展到团队和平台层

## 为什么“控制面放在哪”这么重要

因为用户最终接触到的，不只是能力，还有秩序。

如果一个系统的秩序主要隐含在运行时惯例里，那它往往更像一个有纪律的工作伙伴。  
如果一个系统的秩序更多被显式结构化表达，那它往往更像一个可编排、可治理的平台接口。

两者并没有绝对高下，但适用场景会不同：

- 偏个人流畅体验时，运行时一体化设计常常更顺手
- 偏组织治理和结构扩展时，显式控制层往往更有优势

## 我对这部分的点评

我觉得这本比较书最成熟的地方，就是它没有把差异写成“风格不同”这么抽象的话，而是追问到了控制面。

因为只有追问到这里，你才会真正明白：

- 为什么两个系统都能工作，却让人感觉完全不同
- 为什么同样是 agent coding，有的更像一体化 runtime，有的更像结构化平台
- 为什么同样的能力，在不同控制面之下，会形成不同产品性格

## 这篇读完后应该带走什么

1. 比较 AI coding agent，先比控制面，不要先比功能表。
2. 控制面不是 UI 层，而是系统约束真正安放的位置。
3. `Claude Code` 和 `Codex` 的差异，本质上是秩序组织方式的差异。
