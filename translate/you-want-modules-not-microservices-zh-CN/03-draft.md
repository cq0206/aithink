# 你真正想要的是模块，而不是微服务

原文：<https://blogs.newardassociates.com/blog/2023/you-want-modules-not-microservices.html>

副标题：为什么所有人都在谈微服务？让我们把这件事拆开来看。  
发布日期：2023-01-02  
作者：Ted Neward

## 太长不看

软件架构有时候真的很难。行业里总会不断冒出某个新概念，很快就被包装成“现在大家都该这么做”的主流方案，却几乎不给上下文，也不给任何细腻的边界条件。整个行业又总是急着想改善自己的架构，于是几乎不加迟疑地把它一把抓起。微服务只是这一轮风潮中的最新代表，而现在，是时候把它拆开来，看看问题真正的根到底在哪里了。

## 在微服务的核心，我们总被告知会看到……

……一大堆好东西（TM）！

- “可扩展性”：代码可以被拆成更小的部分，并且可以独立开发、测试、部署和更新。
- “专注”：开发者可以专注于解决业务问题和业务逻辑。
- “可用性”：后端数据必须始终对各种设备保持可用。
- “简洁性”：它让大型企业级应用的开发变得更简单。
- “响应性”：它让分布式应用能够随着事务负载变化而扩缩。
- “可靠性”：通过复制服务器组，避免单点故障；某个部分坏了，系统仍能继续运行，并在故障后恢复。

这些话听起来是不是都很耳熟？有意思的是，这六句引文里，有两句来自微服务文献（博客、论文之类），有两句来自二十年前的 EJB 文献，还有两句来自 Oracle Tuxedo——那是四十多年前的技术。你能分得清哪句来自哪一派吗？

我们这个行业有一个老毛病：总喜欢把同一套 hype 话术翻来覆去地重新包装。

## 在微服务热潮里，我们又看到了 10 个冲进去的理由

某家公司的一篇博客列出了拥抱微服务的 10 个理由：

1. 它促进大数据最佳实践。
2. 它相对容易构建和维护。
3. 它能带来更高质量的代码。
4. 它能简化跨团队协作。
5. 它能实现实时处理。
6. 它有助于快速增长。
7. 它能支持更多输出形式。
8. 它让应用生命周期中的更新评估更容易。
9. 它能实现扩展。
10. 它有大量流行工具可用。

听起来是不是非常熟悉？现在我们不妨换个角度，用“前人早就做过什么”这件事来逐条看看。

### 逐条拆解

1. “促进大数据最佳实践”这件事，并不是微服务独有。管道-过滤器架构（pipes-and-filters）从 70 年代就存在了，而 Unix 早就提出过：每个程序只做好一件事，并且把输出设计成可以成为另一个程序的输入。
2. “容易构建和维护”？这和 Unix 哲学说的是一回事。
3. “提高代码质量”？如果聚焦于一小块问题有助于提升质量，那依然只是 Unix 哲学。
4. “简化跨团队协作”就更有意思了。它暗示 SOA（面向服务架构）天然就意味着重量级通信协议。是指 JSON over HTTP 也算“重量级”吗？还是说他们默认所有 SOA 都一定得带着 SOAP、WSDL、XML Schema 和一整套 WS-* 规范？讽刺的是，微服务本身并没有任何地方阻止你使用这些“重量级”协议；甚至有些微服务鼓吹使用 gRPC，而这种二进制协议反倒更像 CORBA 时代的 IIOP。
5. “支持实时处理”？实时处理早就是个老概念了。很多系统确实会用发布-订阅或事件总线，但这件事根本不需要微服务才做得到。
6. “促进快速增长”？“复用模块化架构”听起来像不像几十年来几乎所有技术都喊过的口号？语言喊过，库喊过，框架喊过。我真想有一天看到一种新技术直接说：去它的复用，我们根本不在乎这个。
7. “支持更多输出形式”？“不同受众以不同方式消费数据集”——这听起来跟报表系统首页文案也没什么两样。
8. “更容易评估生命周期中的更新”？把“机器学习”“高级分析”“模型评估”这些动词堆在一起，不代表这句话就真的有内容。
9. “它能扩展”？有趣极了，这话当年也同样说给 EJB、事务中间件（比如 Tuxedo）以及大型机听过。
10. “有很多工具可用”？每一轮技术热潮发展到一定阶段，工具当然都会冒出来。以前有 CASE 工具，后来有 UML，现在换了一套名字而已。

如果你仔细看，会发现这 10 条里至少有一半都围绕着同一个中心思想：把代码和数据拆成小而独立的块，各自版本化，通过公共输入输出拼成更大的系统。简直就像……

## 在微服务的核心，我们看到的是……

……模块。

没错，就是那个朴素得不能再朴素的“模块”。从 20 世纪 70 年代开始，它就是大多数编程语言的核心概念。再往前其实也有，只不过当年的语言没把它当成一等公民。

在 CLR 上你叫它 assembly；在 JVM 上你叫它 JAR 或 package；在操作系统层面你叫它动态链接库（Windows 上的 DLL，Unix 上的 `so` 和 `a`，macOS 上则是各种 Framework）。但从概念上讲，它们全都是模块。

它们内部格式各不相同，但承担的基本职责是一样的：作为一个能够独立构建、独立管理、独立版本化、独立部署、并且可复用的代码单元存在。

作者随后引用了 David Parnas 1971 年那篇奠基性论文《On the Criteria To Be Used in Decomposing Systems into Modules》中的定义，说明“把系统清楚地分割成定义明确、接口清晰、可单独测试和维护的模块”这件事，早在五十多年前就已经是计算机科学的基本功了。

换句话说，微服务被宣传出来的很多好处，本质上其实只是模块化的好处，而这些好处，我们已经拥有了半个世纪。

那么，为什么围绕微服务会有这么大的喧嚣？

因为微服务从来真正讲的都不是微服务，也不是服务，甚至都不是分布式系统。

## 在微服务的核心，我们真正应该看到的是……

……组织上的清晰度。

Amazon 这类最早公开讨论微服务的公司，真正想推动的并不只是某种“架构原则”，而是一个想法：让开发团队尽可能独立，尽可能少地被别人卡住。

要不要等 DBA 团队改 schema？QA 团队要先拿到一个 build 才能开始测？基础设施团队是不是还没把服务器准备好？UX 团队是不是还在做原型？

作者在这里故意来了一串拟声词，意思很简单：这些原本会卡住团队前进的依赖，被整个开发团队一口“吞”进来了。团队开始拥有那些过去分散在不同职能中的职责和控制权。

也就是说，团队必须成为一个小型化的 IT 组织缩影，内部涵盖分析、开发、设计、测试、数据管理、部署、运维等多种能力。

这当然有代价。要么你得把各种不同技能的人拼成一个团队，要么你要求每个人都具备完整技能组合，也就是所谓的“全栈开发者”。这会让招聘难度直线上升。与此同时，团队还要为自己的生产事故负责，因此 on-call、薪酬和法律责任都会一起跟着过来。

但如果这些问题都处理好了，团队就确实能更加独立地构建自己的交付物，除了时间和键盘敲击速度之外，不再受太多别的因素束缚。

至少理论上是这样。

## 在微服务的核心，我们经常会看到……

……分布式计算谬误。

如果你不熟悉这组谬误，它最早由 Peter Deutsch 在 Sun 向同事们提出，后来又在 1994 年 Ann Wolrath 和 Jim Waldo 的经典论文《A Note on Distributed Computing》中被重新强调。它们本质上都在说一件事：

把分布式系统真正做好——无论你说的“好”是性能、可靠性、扩展性，还是别的什么——都非常难。

当我们把系统拆成运行在同一台机器、同一个操作系统节点里的内存模块时，跨进程或跨库边界传递数据的成本，即便在五十年前都相当低。

但一旦这些数据要跨网络传输——这正是大多数微服务会做的事情——通信延迟就会陡增好几个数量级。这个问题不是靠多加几台机器就能“扩掉”的；很多时候，节点越多，问题反而越复杂。

当然，你可以说：那就把这些微服务放在同一台机器上，比如用 Docker Compose 或 Kubernetes，在一组虚拟机或容器里托管它们。

可这只是把问题换了个样子。数据仍然要穿过虚拟网络栈，上下走一遍分层协议；你依旧承担了进程边界、网络边界和虚拟化带来的延迟，而且单节点可靠性问题也没有消失。

更糟糕的是，当你刚开始认真面对分布式计算谬误时，往往还会撞上另一组相邻但不同的问题：企业计算谬误。

## 在微服务的核心，我们真正需要的是……

……重新思考我们到底需要什么。

如果你需要把问题拆成彼此独立的实体，那有很多办法可以做到。你可以使用运行在 Docker 容器里的独立进程，也可以使用部署在应用服务器中的独立模块，只要它们遵守某套标准化 API 约定；当然，还有很多别的方案。

这不是一个“必须抛弃过去二十年所有技术积累”的技术问题。Servlet、ASP.NET、Ruby、Python、C++，甚至——作者故意挖苦一句——也许连 Perl 都能做。

真正关键的是，要建立一块大家都理解的架构底座：清楚的集成方式、清楚的通信约定，以及足够稳定的协作规则。

如果你真正想解决的是“开发团队被依赖阻塞”的问题，那就先去看这些依赖本身，然后和相关团队合作，判断哪些能力可以真正纳入你们团队自己的掌控范围。

如果组织不愿意正式拆掉那种按技能划分的组织结构——比如数据库组、基础设施组、QA 组和开发组彼此平级——那至少也该和高层协商出一种“虚线汇报”或矩阵式的协作方式，让各个职能里的一些成员以矩阵形式被并入同一个团队。

但最重要的是，确保这个团队对自己到底要构建什么有极其清晰的认识，而且能把其服务、微服务或模块的核心目标，清楚地讲给任何一个路过的陌生人听。

关键在于：给团队方向和目标，给团队完成任务的自主权，也给团队把事情真正做成的使命感。

说到底，整件事最终可以归结为两个问题：

1. 软件应该如何做模块化拆分。
2. 团队应该如何获得足够的组织自主性。

这两件事当然有关，但它们真的、真的、真的不是同一回事。
