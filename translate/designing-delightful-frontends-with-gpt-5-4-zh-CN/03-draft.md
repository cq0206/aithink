# 用 GPT-5.4 设计令人眼前一亮的前端

原文：<https://developers.openai.com/blog/designing-delightful-frontends-with-gpt-5-4>

发布日期：2026-03-20  
作者：Brian Fioca、Alistair Gillespie、Kevin Leneway、Robert Tinn

GPT-5.4 在 Web 开发上的表现比此前的版本更强，生成出来的前端不仅更好看，也更敢做、更有野心。尤其值得注意的是，我们在训练 GPT-5.4 时，专门强化了它的 UI 能力和对图像的使用能力。只要给出合适的引导，这个模型就能做出达到生产可用水准的前端界面，其中包括细腻的小设计、精心打磨的交互，以及优质的视觉素材。

网页设计的结果空间非常大。真正优秀的设计，往往是在克制与创造之间取得平衡：既借鉴那些经得起时间考验的模式，又能带来一点新的东西。GPT-5.4 已经学习了这一整套宽广的设计谱系，也理解网站可以通过很多不同方式构建出来。

当提示词写得不够具体时，模型往往会回退到训练数据里最常见的高频模式。其中一部分确实是成熟惯例，但也有不少只是被过度重复的习惯，而这些恰恰是我们希望避免的。结果通常会“能用、像样”，但也容易滑向结构泛化、视觉层级薄弱、设计选择不够贴合你脑海中真正想要的效果。

这篇指南讲的，就是如何用一套务实的方法，把 GPT-5.4 引导到你真正想做出的那种设计上。

## 模型能力的提升

虽然 GPT-5.4 在很多维度上都有提升，但就前端工作而言，我们尤其聚焦了三个很实际的改进方向：

- 在整个设计过程中更强的图像理解能力
- 能生成功能更完整的网站和应用
- 更善于使用工具去检查、测试并验证自己的工作

### 图像理解与工具使用

GPT-5.4 在训练时已经原生学会了使用图像搜索和图像生成工具，因此它可以把视觉推理直接纳入自己的设计过程。为了取得更好的结果，建议你先让模型生成一份情绪板，或者先给出几套视觉方向，再从中选定最后使用的素材。

你可以通过明确描述图像应当呈现的特征，来把模型引向更强的视觉参考，比如风格、配色、构图或整体氛围。你也可以在提示词里进一步说明：优先复用已经生成过的图片、在需要时调用图像生成工具产出新视觉，或者在任务确有需要时引用指定的外部图片。

> 默认优先使用任何已经上传或预先生成好的图片。否则，使用图像生成工具来创建视觉效果出色的图像素材。除非用户明确要求，否则不要引用或链接网页上的图片。

### 功能性提升

这个模型也接受过训练，以便开发出更完整、在功能上更健全的应用。你可以预期，在长流程任务里，它会比过去更可靠。过去你可能觉得一两轮对话根本做不出来的游戏或复杂用户体验，现在已经变成现实。

### 计算机使用与验证

GPT-5.4 是我们首个针对“计算机使用”训练的主线模型。它可以原生地在界面中导航；再结合 Playwright 这类工具，它就能以迭代方式检查自己的成果、验证行为，并不断修正实现，从而支持更长链路、更加自主的开发工作流。

你可以通过官方发布视频看到这些能力的实际表现。

对于前端开发来说，Playwright 尤其有价值。它允许模型去检查渲染后的页面、测试多个视口、走完整个应用流程，并发现状态或导航方面的问题。只要你给模型配上 Playwright 这样的工具或技能，GPT-5.4 产出精致、功能完整界面的概率就会显著提高。再加上更强的图像理解能力，它还可以从视觉上反查自己的工作，确认结果是否与参考 UI 一致。

## 实用建议速览

如果你只打算从这篇文章里带走几条最关键的做法，可以先记住下面这些：

1. 一开始先选用较低的推理强度。
2. 提前定义好设计系统和约束，例如字体、色盘、布局。
3. 提供视觉参考或情绪板，例如附上一张截图，给模型明确的视觉护栏。
4. 预先定义页面叙事或内容策略，指导模型生成内容。

下面是一段可以直接拿来起步的提示词。

## 前端任务

在做前端设计任务时，避免生成那种泛泛而谈、堆得很满的通用布局。

**请遵守以下硬规则：**

- 单一构图：首屏视口必须被读成一个完整构图，而不是仪表盘拼盘（除非它本来就是仪表盘）。
- 品牌优先：在品牌型页面中，品牌名或产品名必须是 Hero 级别的视觉信号，而不能只是导航里的一个文字或小眉题。任何标题都不应盖过品牌本身。
- 品牌测试：如果去掉导航之后，首屏看起来还能像是另一个品牌的页面，那就说明品牌表达太弱。
- 字体：使用有表达性、带明确目的性的字体，避免默认字体栈（Inter、Roboto、Arial、system）。
- 背景：不要只依赖纯色平面背景；用渐变、图像或细微纹理来建立氛围。
- 只用全幅 Hero：在落地页和宣传型界面里，Hero 图片默认应当是一个占据主导地位、边到边铺满的视觉平面或背景。除非既有设计系统明确要求，否则不要使用内嵌式 Hero 图片、侧栏 Hero 图片、圆角媒体卡片、拼贴式宫格或漂浮图块。
- Hero 预算：首屏通常只应包含品牌、一个标题、一句简短辅助文案、一组 CTA，以及一张主导性图片。不要在首屏塞入统计、时间表、活动列表、地址区块、促销信息、“本周推荐”、元信息行或次级营销内容。
- 不要 Hero 叠层贴片：不要把独立标签、漂浮徽章、促销贴纸、信息小胶囊或提示框压在 Hero 媒体上。
- 卡片：默认不要卡片。Hero 中绝不能使用卡片。只有当卡片本身就是用户交互容器时，卡片才是允许的。如果去掉边框、阴影、背景或圆角之后，并不会损害交互或理解，那它就不该是卡片。
- 每个 section 只做一件事：每个 section 只能有一个目的、一个标题，通常再配一句简短辅助说明。
- 真实的视觉锚点：图像应当展示产品、场所、氛围或上下文。装饰性渐变和抽象背景不能算作主要视觉概念。
- 降低杂乱感：避免胶囊标签堆、统计条、图标排排站、带框促销块、日程碎片和多个彼此竞争的文本块。
- 动效应当用来建立存在感与层级，而不是制造噪音。对于以视觉驱动的作品，至少交付 2 到 3 个有明确意图的动画。
- 色彩与整体观感：先决定清晰的视觉方向，定义 CSS 变量，避免默认的“紫色配白底”套路。不要天然偏向紫色，也不要天然偏向暗黑主题。
- 确保页面在桌面端和移动端都能正常加载。
- 如果使用 React，在团队已有相关做法时，优先考虑 `useEffectEvent`、`startTransition`、`useDeferredValue` 这类现代模式。除非项目本来就在用，不要默认加入 `useMemo` 或 `useCallback`；遵循项目的 React Compiler 约定。

例外：如果任务发生在已有的网站或设计系统内，就必须保留现有模式、结构和视觉语言。

## 如何做出更好的设计

### 先从设计原则开始

先定义约束，例如：只允许一个 H1 标题、页面 section 不超过 6 个、最多两种字体、只用一个强调色、折叠线以上只保留一个主要 CTA。

### 提供视觉参考

参考截图或情绪板可以帮助模型推断页面的版式节奏、字号层级、留白系统和图像处理方式。文章里给出的例子，就是 GPT-5.4 在 Codex 中先为用户生成一份情绪板，再由用户据此确认视觉方向。

### 把页面组织成一个叙事

典型的营销页面结构通常是：

1. Hero：建立身份和承诺
2. 支撑性图像：展示场景或环境
3. 产品细节：说明你到底提供什么
4. 社会证明：建立可信度
5. 最终 CTA：把兴趣转化为行动

### 明确要求遵守设计系统

应当鼓励模型在构建早期就先建立清晰的设计系统。你可以定义核心设计令牌，例如 `background`、`surface`、`primary text`、`muted text` 和 `accent`，再定义字体角色，例如 `display`、`headline`、`body`、`caption`。这样能帮助模型在整个应用里生成一致、可扩展的 UI 模式。

对大多数 Web 项目来说，从 React + Tailwind 这类熟悉的技术栈起步通常效果不错。GPT-5.4 在这些工具上的表现尤其强，因此更容易快速迭代并做出打磨度较高的结果。

动效和分层 UI 元素会引入额外复杂度，尤其是在固定或漂浮组件与主要内容互相作用时。处理动画、遮罩层或装饰图层时，最好明确写出有利于安全布局的指导语。例如：

> 让固定或漂浮的 UI 元素在各种屏幕尺寸下都不要遮挡文本、按钮或其他关键内容。把它们放在安全区域中，在适当情况下置于主要内容之后，并保持足够间距。

### 降低推理强度

对于比较简单的网站，推理越多并不一定越好。实践中，较低和中等推理强度往往反而能带来更强的前端结果：模型会更快、更聚焦、不那么容易过度思考；而当你要做更有野心的设计时，又仍然保留了进一步调高推理强度的空间。

### 让设计扎根于真实内容

给模型提供真实文案、真实产品上下文，或者明确的项目目标，是提升前端结果最简单有效的方法之一。这些上下文能帮助它选择更合适的网站结构、形成更清晰的 section 级叙事，并写出更可信的内容，而不是回退到千篇一律的占位文案。

## 用 Frontend Skill 把这些方法整合起来

为了帮助开发者在通用前端任务中更充分地发挥 GPT-5.4 的能力，OpenAI 还准备了一个专门的 `frontend-skill`。它会在结构、审美和交互模式上给模型更强的引导，让它从一开始就更容易生成经过打磨、有明确意图、并且真正令人愉悦的设计。

### Frontend skill

```md
---
name: frontend-skill
description: Use when the task asks for a visually strong landing page, website, app, prototype, demo, or game UI. This skill enforces restrained composition, image-led hierarchy, cohesive content structure, and tasteful motion while avoiding generic cards, weak branding, and UI clutter.
---
```

# Frontend skill

当任务质量主要取决于艺术指导、层级关系、克制感、图像与动效，而不是组件数量时，就使用这个 skill。

目标是：交付那种看起来有明确意图、质感高级、符合当下审美的界面。默认朝“获奖级构图”靠拢：一个大想法、强视觉图像、稀疏文案、严格间距，以及少量但难忘的动效。

## 工作模型

在开始构建前，先写清楚三件事：

- visual thesis：用一句话描述氛围、材质和能量感
- content plan：hero、support、detail、final CTA
- interaction thesis：2 到 3 个会改变整页气质的动效想法

每个 section 都只做一件事、只承载一个主导性视觉概念，并且只留下一个主要信息点或行动目标。

## 优雅的默认值

- 先做构图，不要先堆组件。
- 优先使用全幅 Hero 或全画布级的视觉锚点。
- 让品牌名或产品名成为页面中最响亮的文字。
- 文案要短到几秒内就能扫完。
- 在添加边框、装饰和界面“外壳”之前，先用留白、对齐、尺度、裁切和对比来解决问题。
- 把系统控制住：最多两种字体，默认只用一个强调色。
- 默认使用非卡片式布局。改用 section、分栏、分割线、列表和媒体模块。
- 把首屏当成一张海报，而不是一份文档。

## 落地页

默认顺序：

1. Hero：品牌或产品、承诺、CTA 和一个主导性视觉
2. Support：一个具体功能、优惠或证据点
3. Detail：氛围、工作流、产品深度或故事
4. Final CTA：转化、开始、访问或联系

Hero 规则：

- 只能有一个整体构图。
- 使用全幅图片或占据主导地位的视觉平面。
- 经典全幅规则：在品牌型落地页中，Hero 本身必须边到边铺满，不要继承页面左右边距、外框容器或共享的最大宽度；只约束内部文字与操作列。
- 品牌第一，标题第二，正文第三，CTA 第四。
- 默认不要 Hero 卡片、统计条、Logo 云、胶囊标签大杂烩或漂浮仪表盘。
- 标题在桌面端大约控制在 2 到 3 行，移动端要做到一眼可读。
- 文字列要保持窄，并停留在图片中相对平静的区域。
- 所有叠加在图像上的文字都必须保持足够对比度和清晰的点击目标。

如果把首屏中的图片去掉之后，这个首屏依然成立，那说明图片不够强。如果把导航隐藏后品牌就消失了，那说明层级关系不够强。

视口预算：

- 如果首屏包含 sticky 或 fixed 的头部，这个头部也要算进 Hero 预算。头部与 Hero 的合计内容，必须能在常见桌面与移动端尺寸下放进初始视口。
- 如果使用 `100vh` 或 `100svh` 的 Hero，记得减去持久存在的界面元素高度（例如 `calc(100svh - header-height)`），或者把头部做成覆盖式，而不是在正常文档流中向下堆叠。

## 应用型界面

默认采用类似 Linear 的克制风格：

- 平静的界面层级
- 强字体与强间距
- 少量颜色
- 信息密集但可读
- 极少的界面装饰
- 只有当卡片本身就是交互容器时才使用卡片

对于应用 UI，应围绕下面这些结构组织：

- 主工作区
- 导航
- 次级上下文或检查面板
- 一个清晰的强调色，用于动作或状态

避免下面这些做法：

- 仪表盘式卡片马赛克
- 每个区域都用厚边框框起来
- 在常规产品 UI 后面放装饰性渐变
- 多个彼此竞争的强调色
- 对扫读没有帮助的装饰性图标

如果一个面板去掉卡片处理后仍然不影响理解，那就把卡片去掉。

## 图像

图像必须承担叙事工作。

- 对品牌、场所、编辑型页面和生活方式产品，至少使用一张强而真实的图片。
- 优先使用在真实场景中的摄影，而不是抽象渐变或伪 3D 物体。
- 选择或裁切那些能为文字留出稳定明暗区域的图片。
- 不要使用带有明显招牌、Logo 或会和 UI 争夺注意力的排版杂讯图片。
- 不要生成自带 UI 边框、分屏、卡片或面板的图片。
- 如果需要表达多个时刻，就使用多张图片，不要拼成一张拼贴。

首屏必须有真实的视觉锚点。光有装饰纹理是不够的。

## 文案

- 使用产品语言，而不是设计评论式语言。
- 让标题承担核心意义。
- 辅助文案通常只需要一句短句。
- 删除各个 section 之间重复表达的内容。
- 不要把提示词语言或设计评论直接写进 UI。
- 给每个 section 分配一个职责：解释、证明、深化，或转化。

如果删掉 30% 的文案后页面更好，那就继续删。

## 产品 UI 的功能型文案

当你做的是仪表盘、应用界面、后台工具或运营工作台时，默认使用功能型文案，而不是营销型文案。

- 优先传达方向感、状态与动作，而不是承诺、情绪或品牌腔调。
- 从真正的工作表面开始：KPI、图表、筛选器、表格、状态或任务上下文。如果用户没有明确要求，不要主动加 Hero 区。
- section 标题应直接说清楚这个区域是什么，或者用户能在这里做什么。
- 好例子："已选 KPI"、"计划状态"、"搜索指标"、"头部细分"、"最近同步"
- 除非用户明确要求，否则避免在产品界面中使用充满愿景感的 Hero 句子、比喻性文案、活动式语言和高管摘要横幅。
- 辅助文案应当用一句话说明范围、行为、新鲜度或决策价值。
- 如果一句话看起来像能出现在首页 Hero 或广告里，那就继续改，直到它听起来像真正的产品 UI 文案。
- 如果某个 section 不能帮助用户操作、监控或做决策，那就删掉。
- 最后的自检问题是：如果一个操作者只扫一遍标题、标签和数字，他是否能立刻明白这个页面？

## 动效

动效应该用来建立存在感和层级，而不是制造噪音。

对以视觉驱动的作品，至少交付 2 到 3 个有明确意图的动效：

- 一个首屏入场序列
- 一个滚动联动、sticky 或景深效果
- 一个 hover、revea 或布局过渡，用来强化可感知性

如果条件允许，优先用 Framer Motion 来实现：

- section 出场动画
- shared layout transitions
- 基于滚动的透明度、位移或缩放变化
- sticky 叙事
- 推动叙事的轮播，而不是纯粹填充空间的轮播
- 菜单、抽屉和模态层的出现效果

动效规则：

- 在快速录屏里也能看出存在感
- 在移动端足够流畅
- 快速但克制
- 整页风格一致
- 如果只是装饰，就删掉

## 硬规则

- 默认不要卡片。
- 默认不要 Hero 卡片。
- 当 brief 明确要求全幅时，不要做有边框盒子感或居中窄列式 Hero。
- 每个 section 不要有超过一个主导性想法。
- 不要让一个 section 靠一堆细碎小 UI 装置才能说清楚自己。
- 在品牌型页面里，不要让标题盖过品牌。
- 不要有填充性文案。
- 除非文字停留在一侧平静、统一的区域，否则不要用 split-screen Hero。
- 没有充分理由时，不要超过两种字体。
- 除非产品本身已经有很强的系统，否则不要超过一个强调色。

## 要拒绝的失败结果

- 首屏就是一套千篇一律的 SaaS 卡片网格
- 图片很美，但品牌存在感很弱
- 标题很强，但没有清晰行动
- 图片背景太忙，压住了文字
- 不同 section 重复表达同一种情绪
- 轮播没有叙事目的
- 应用 UI 由一摞堆叠卡片拼出来，而不是由布局组织起来

## 快速自检

- 用户能否在第一屏就明确识别出品牌或产品？
- 是否存在一个强有力的视觉锚点？
- 只扫标题，是否也能理解整页？
- 每个 section 是否都只做一件事？
- 卡片真的有必要吗？
- 动效是否提升了层级或氛围？
- 如果去掉所有装饰性阴影，设计是否仍然显得高级？
